# Naprawa dwóch błędów UI

## Bug 1: Mignięcie przy zmianie motywu

### Przyczyna
`globals.css:132,136,139` definiują `transition: background-color 0.2s ease` na `html`, `body` i `.bg-card/.bg-muted/.bg-background`. Gdy `ThemeProvider.tsx:35-36` natychmiast przełącza klasę `.light` na `<html>`, zmienne CSS snapują instant, ale transition animuje kolory przez 200ms → widoczne "mignięcie".

### Rozwiązanie
W `ThemeProvider.tsx`, w funkcji `applyTheme` (linia 32-37), przed przełączeniem klasy:
1. Wstrzyknij `<style>` z `* { transition: none !important; }`
2. Wykonaj classList remove/add
3. Wymuś repaint: `document.documentElement.offsetHeight`
4. Usuń `<style>` po krótkim opóźnieniu (next frame via `requestAnimationFrame`)

To eliminuje flash bez usuwania CSS transition (przydatnych przy hover/interakcjach).

### Plik: `src/components/ThemeProvider.tsx`
Zmodyfikuj `applyTheme` (linie 32-37):
```tsx
const applyTheme = useCallback((t: Theme) => {
  const resolved = t === 'system' ? getSystemTheme() : t;
  setResolvedTheme(resolved);

  // Tymczasowo wyłącz transitiony
  const style = document.createElement('style');
  style.textContent = '* { transition: none !important; }';
  document.head.appendChild(style);

  document.documentElement.classList.remove('light');
  if (resolved === 'light') document.documentElement.classList.add('light');

  // Wymuś repaint, potem usuń style
  document.documentElement.offsetHeight;
  requestAnimationFrame(() => {
    document.head.removeChild(style);
  });
}, []);
```

## Bug 2: Dane w topbar pokazują 0/'-' po przeładowaniu

### Przyczyna
- `useStats.ts:35` i `useJobs.ts:78`: `useState(undefined)` — cache startuje jako `undefined`
- localStorage jest czytany w `useEffect` (linie 37-39, 80-82), który Execute PO pierwszym renderze
- Pierwszy render → `total=0`, `stats=null`, `lastSync='-'`, `bySource.length=0`
- `AnimatedNumber.tsx:6`: `useState(0)` — animacja zaczyna od 0, co powoduje widoczne odliczanie

### Rozwiązanie (część A): Eager initialization z localStorage
W obu hookach zmień `useState(undefined)` na `useState(() => loadCache())`. React wywołuje initializer tylko przy pierwszym renderze. Na serwerze `loadCache()` zwraca `undefined` (guard `typeof window`), więc jest SSR-safe.

Usuń `useEffect` ładowający cache (niepotrzebny).

#### `src/hooks/useStats.ts` (linia 35, 37-39):
```tsx
// PRZED:
const [cached, setCached] = useState<Stats | undefined>(undefined);
useEffect(() => { setCached(loadCache()); }, []);

// PO:
const [cached] = useState<Stats | undefined>(() => loadCache());
```

#### `src/hooks/useJobs.ts` (linia 78, 80-82):
```tsx
// PRZED:
const [cached, setCached] = useState<JobsResponse | undefined>(undefined);
useEffect(() => { setCached(loadCache(key)); }, [key]);

// PO:
const [cached] = useState<JobsResponse | undefined>(() => loadCache(key));
```

### Rozwiązanie (część B): AnimatedNumber snap do pierwszej wartości
W `AnimatedNumber.tsx`, dodaj ref `hasInitialized`. Gdy `value` po raz pierwszy zmieni się na niezerowe → snap directo do value bez animacji.

#### `src/components/AnimatedNumber.tsx`:
```tsx
const hasInitialized = useRef(false);

useEffect(() => {
  const from = prevRef.current;
  const to = value;
  if (from === to) return;

  // Pierwsza prawdziwa wartość — snap directo
  if (!hasInitialized.current && to !== 0) {
    setDisplay(to);
    prevRef.current = to;
    hasInitialized.current = true;
    return;
  }

  hasInitialized.current = true;
  // ... reszta animacji
}, [value, duration]);
```
